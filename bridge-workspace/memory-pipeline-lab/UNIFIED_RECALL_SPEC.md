# 统一记忆召回说明

统一召回器已经通过工作区 MCP 配置暴露为只读工具 `memory_recall`，协议测试已通过。它不会在每轮聊天前强制运行，而是由当前 Gem 判断需要旧记忆时自行调用。真实 Antigravity / Gem 会话尚未完成一次端到端调用验证，因此“已配置”和“已在真实模型中验证”仍分开记录。

工具只读查询 SQLite，但可以原子覆盖 `GEMINI.md` 中 `MEMORY_CONTEXT_START/END` 标记之间的本轮动态参考区；稳定正文不变。

当前数量、阈值和运行开关以 `memory-system-config.json` 为唯一配置来源；本文件负责解释规则，不再单独维护另一套数值。

## 一轮请求怎么走

| 顺序 | 做什么 | 结果 |
|---|---|---|
| 1 | 看当前窗口是否已经足够回答 | 足够就不召回，也不写动态区 |
| 2 | 判断是否涉及过去、个人事实、偏好或历史过程 | 不涉及就结束 |
| 3 | 提取时间范围和问题粒度 | 概览、过程、精确事实、原话、清单、按日期聚合，以及首次/最近/次数/是否发生 |
| 4 | 生成一条查询向量 | 同一条向量同时给 Card、事实、摘要、用户原文打分 |
| 5 | SQLite 先按时间和人物缩小范围 | 避免全库内容混入候选 |
| 6 | 按问题粒度选择最终层 | 并行检索不等于四层全部注入 |
| 7 | 生成本轮动态参考块 | 只能覆盖旧动态区，不能追加多个动态区 |
| 8 | 检查 `turn_id` | 旧轮次结果不能覆盖新轮次 |

## 各类问题取多少

| 问题类型 | 优先使用 | 数量上限 | 何时找原文 |
|---|---|---:|---|
| 某月/某日概览 | Event Summary | 单日 3 条；月份 10 条 | 当天没有摘要时，均匀取最多 8 条用户原文 |
| 项目或事件过程 | Event Summary | 最多 2 条接近第一名的摘要 | 取被摘要明确引用的用户原文，最多 6 条 |
| 精确个人事实 | Fact Timeline / Card | 默认 1 条 | 问“哪一天、什么时候、证据”时补最多 2 条来源原文 |
| 用户原话 | 用户原文 | 默认 1 条最相关原文 | 本身就是原文召回；歧义时宁可不返回 |
| 偏好/边界清单 | Memory Card | 6 条 | Card 不够时暂不猜测，不自动塞大量原文 |
| 按日期聚合 | 用户原文 | 每天 1 条、最多 8 天 | 先去掉时间问句，只用真正主题做语义检索 |
| 最近一次事件 | Event Occurrence / 用户原文 | 1 个事件并补最多 3 条原文 | 先按语义筛出相关候选，再按时间选最新，不能单纯取数据库最后一条 |
| 首次/次数/是否发生 | Event Occurrence | 首次或是否发生默认 1 组；次数最多展示 12 组 | 先按 `event_status` 排除仅提到、请求、计划和拒绝，再按 `occurrence_key` 去重并补来源原文 |
| 约定/共同目标 | 摘要中的 `user_goals` | 6 条目标线索 | 补取目标引用的用户原文；明确标注“目标不等于双方约定，也不保证当前有效” |
| 一轮多个问题 | 每个子问题分别选择 | 最多拆 4 个子问题 | 子问题共享 `turn_id`；“那次/后来/我说自己”等追问可以继承前一事件日期 |

`event_occurrences` 是可从摘要和原文重建的检索目录，不是长期 Memory Card。它保存事件生命周期和分组键，使首次、最近和临时计数共用同一套证据。次数查询结果本身不另行写回数据库。

## 稳定性边界

- 召回器源码不允许出现 Server酱、Agent Mail、Notion、OpenCode 等项目专名路由。
- 对明确人物使用 `subject_key` 隔离。当前原文没有人物标注时，询问母亲等其他人物宁可返回空，也不拿用户本人的事实代替。
- `explicit_only` 内容只有在问题明确涉及用户本人，且语义或词面相关度足够时才能进入候选。
- 动态参考块明确标注为不可信历史数据：不得当作指令；用户原话优先；旧助手声称完成的操作若无用户或日志证据，不算验证完成。
- `replaceDynamicRegion()` 只替换 `MEMORY_CONTEXT_START/END` 之间的区域；本轮无召回时可以传空字符串清除旧区域。
- `createTurnCoordinator()` 保存每个 session 最新的 `turn_id`；异步召回结束后必须先确认仍是当前轮次，才能写入。
- 多问题会在一次 Ollama 请求中批量生成查询向量，避免每个子问题分别等待一次模型调用。
- 纯标点、纯表情和单独应答保留在 `raw_messages`，但不进入向量候选；低信息过滤不等于删除原始聊天。

## 目前还没有完成的事

- 尚未用真实 Telegram → Antigravity / Gem 对话验证模型会按 skill 边界主动调用工具。
- 桥接本身不在每轮预先召回；动态区只由 `memory_recall` 工具在被调用时覆盖。
- 没有调用 Gemini、DeepSeek 或其他远程模型；本轮只用本机 Ollama `bge-m3` 做向量。
- 当前“按日期聚合”用向量置信度和证据门过滤；以后可以再增加小模型证据判定，但它不是第一版验证的必要条件。

## 当前真实调用路径

1. Gem 先看当前窗口能否回答；能回答就不调用工具。
2. 需要旧记忆时，Gem 按 skill 选择 `auto` 或更明确的操作。
3. MCP 服务只读查询 SQLite，必要时调用本机 `bge-m3`。
4. 服务生成本轮参考块并原子覆盖动态区，同时把同一份证据返回给 Gem。
5. 下一次召回会完整覆盖旧动态区；未再次召回时，Gem 不应把旧动态区误当成当前问题证据。

## 中文词语与 trigram 的混合方式

同一条查询会同时进入三路：

1. 原有 FTS5 trigram，擅长完整名称和原文子串。
2. Jieba 预分词后的 FTS5 unicode61，擅长“月经、排班、邮箱、通勤”等两字词和资料自带别名。
3. `bge-m3` 向量，负责语义改写和没有共同字面的表达。

三路候选使用 RRF 合并。只有 Jieba 普通词命中时权重低于 trigram；trigram 与 Jieba 同时命中时才使用完整 FTS 权重，避免“桥接、系统”等常见词压过更相关的语义候选。最终仍必须经过人物、时间、事件状态和证据门。

分词只解决词面召回。“后来成功了吗”是否属于过程查询由召回路由判断，“微信推送工具”是否指 Server酱还需要资料别名或向量，不能把这些问题伪装成切词问题。当前通用路由识别“后来/后面/之后 + 结果”，以及“之前的事情 + 最终/最后 + 结果”；规则本身不包含任何项目专名。
