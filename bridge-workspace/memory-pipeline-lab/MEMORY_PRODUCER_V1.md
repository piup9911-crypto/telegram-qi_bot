# 自动记忆生产器 V1

这部分解决的不是“怎么召回”，而是“原始聊天怎样安全地变成可召回的摘要、Card、事实线和事件”。

当前 Telegram 桥接已经把新增原文写入正式 SQLite，并把符合条件的批次放入 `memory_processing_jobs`。外部整理模型和后台 worker 调度仍关闭，所以现在只会排队，不会调用 Gemini、DeepSeek 或其他外部模型，也不会自动生成新的 Summary / Card / Fact。

批次、重试和召回参数统一保存在 `memory-system-config.json`，中文可视化总览由 `build-memory-system-config-page.cjs` 生成。

## 一句话流程

`未处理原消息 → 自动凑一小批 → 模型按内容分段 → 证据程序验收 → 一次性写入 SQLite → 刷新文字索引`

固定时间窗不参与分段。队列只限制一次最多送多少消息和字符，真正的边界由对话内容决定；聊天不连续、隔很久并不会因此被强行分成错误事件。

## 自动处理规则

| 情况 | 结果 |
|---|---|
| 寒暄、重复、低价值碎片 | 标为 `raw_only`，仍保留原文，可由全文/向量检索找回 |
| 某段发生了什么 | 写 `event_summaries`，分为 `index` 或 `detailed` |
| 用户稳定偏好、边界、长期模式 | 有用户原话证据才写 `memory_cards` |
| 会变化的用户状态 | 写 `fact_timelines` + `fact_events`；旧值保留，当前指针移到新值 |
| 某次请求、开始、完成、失败 | 写 `event_occurrences`，便于日期查询和临时计数 |
| 助手自己的推测 | 不能成为用户 Card 或用户事实 |

## 稳定与正确怎么保证

1. 模型不能直接写数据库，只能返回候选 JSON。
2. 每一批消息必须被连续、无遗漏、无重复地分段。
3. 每张 Card 和每条用户事实必须带原消息 ID 和原文中的逐字引用。
4. 程序检查“引用是否真的存在、说话人是不是用户、证据是否属于本段”。任一项不对，整批不写。
5. `sensitive` 内容只能设置为 `explicit_only`，避免普通相关召回自动带出。
6. 相同批次、相同 Card、相同事实证据和相同事件都使用稳定键；任务重跑不会复制一份。
7. 临时超时进入 `retry_wait`，指数退避后自动重试；证据造假属于永久失败，不会盲目重试后写入。
8. 同一会话同一类后台任务只允许一个处于活动状态，避免并发改乱事实线。

## 后台任务表新增字段

没有增加新的记忆业务表，只给 `memory_processing_jobs` 增加审计字段：

| 字段 | 作用 |
|---|---|
| `input_hash` | 确认这一批输入是否已经处理过 |
| `output_json` | 保存模型原始候选输出 |
| `validation_json` | 保存程序验收结果和具体错误 |
| `write_result_json` | 保存实际新增、复用、冲突的数量 |
| `processor_version` | 记录使用的生产规则版本 |
| `committed_at` | 记录真正写入完成时间 |

因此不需要人工逐条审计，但出了问题可以查清“模型当时说了什么、为什么被接受、最终写了什么”。

## 当前已经验证

运行：

```powershell
node run-memory-producer-offline-lab.cjs
```

测试只使用虚构对话，并在 `memory-producer-offline.sqlite` 副本中运行。它会验证：正常生成、错误证据拦截、重复运行幂等、临时失败重试、同会话并发阻止、文字索引刷新。正式桥接只复用了已经验证过的入队逻辑，不会执行这份测试数据。

## 尚未启用的部分

- 外部模型适配器仍关闭。以后可把同一份 `buildProducerRequest()` 交给 Gemini Flash 或 DeepSeek，再把 JSON 送回现有验收器；模型无权绕过验收器。
- 后台 worker 的空闲调度器仍关闭。启用时只需消费现有任务队列，不修改当前聊天回复和召回路径。
- 向量缓存会在现有统一召回器下次启动时按内容指纹自动刷新；词索引在写入后立即刷新。
