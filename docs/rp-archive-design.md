# RP Archive Snapshot Design

This document defines the next-step archive model for `source_type = "telegram_rp"`.
It is intentionally design-only for now: the first runtime milestone only needs
`/api/rp/generate` and prompt preview to work.

## Current Record Shape

Active RP chats live in the existing Gem Chat Records state directory:

```text
bridge-state/chats/telegram_rp_<telegram_chat_id>.json
```

The active file already uses immutable `chatId` as the internal identifier and
keeps `display_name` as presentation-only metadata.

## Archive Goals

An RP archive must be restorable without depending on mutable live config rows.
The archive may keep original IDs as references, but restore should default to
the archived snapshots.

Archive files should remain read-only. Continuing from an archive creates a new
active RP session and leaves the original archive unchanged.

## Proposed Archive State

```json
  "archiveId": "archive-20260514-153000",
  "source_type": "telegram_rp",

  "display_name": "Example RP",
  "archivedAt": "2026-05-14T07:30:00.000Z",
  "sessionId": null,
  "history": [],
  "rp_snapshot": {
    "schema_version": 1,
    "source_chat_id": "telegram_rp_7541487750",
    "source_session_id": "active-session-id",
    "character": {
      "id": null,
      "snapshot": null
    },
    "preset": {
      "id": null,
      "snapshot": null
    },
    "persona": {
      "id": null,
      "snapshot": null
    },
    "lorebooks": [],
    "scene_state": {
      "id": null,

```

Each item with an `id` field may refer to a live object, but `snapshot` is the
restore source of truth. For lists, use the same pattern:

```json
{
  "id": "lorebook-id",
  "snapshot": {
    "name": "Archived lorebook",
    "entries": []
  }
}
```

## Proposed APIs

First minimal API surface:

- `POST /api/telegram-rp/:chatId/archive-window`
  - Copies active `history`, `display_name`, `source_type`,
    `telegram_chat_id`, immutable `chatId`, and `rp_snapshot`.
  - Writes an archive under the existing archive directory.
  - Resets the active RP chat to an empty new session.

- `GET /api/telegram-rp/:chatId/archives`
  - Lists read-only RP archives for one RP chat.

- `POST /api/telegram-rp/:chatId/archives/:archiveId/continue`
  - Creates a new active RP chat from the archive snapshot.
  - Does not mutate the archive.
  - Uses a new immutable chat/session id.
  - Default display name is `<archive display_name>（续）`.

## Immutable ID Rule

Never use `display_name` as an identifier. UI actions should pass:

- active RP chat id: `telegram_rp_<telegram_chat_id>` or a future generated
  session id
- archive id: `archive-YYYYMMDD-HHMMSS`
- message ids generated by `messageId(...)`

## Restore Rule

Restore should load from `rp_snapshot` by default. If the live referenced ID
still exists, the UI may offer an explicit "update to live config" action later,
but this should not happen implicitly.

## Current Non-Goals

- No model provider snapshot yet beyond preset placeholder.
- No vector memory or graph data.
- No UI for archive continuation in this milestone.
- No mutation of archived windows.
